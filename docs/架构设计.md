# 裸机工程架构设计思路

## 写在前面

结合 [preview.md](/home/ssw12/projects/STM32F103C8T6-Template/docs/preview.md) 对当前模板工程的分析，可以明确一点：

当前仓库已经具备一个很好的“启动与构建底座”，但它还处在“模板基线”阶段，而不是一个已经完成长期可维护分层设计的裸机工程。

如果目标是把这个模板逐步演进成一个**健壮、可维护、可扩展、适合多人协作**的裸机项目，那么架构设计的重点不只是“把文件夹分出来”，而是要同时解决下面几类问题：

- 代码依赖关系是否清晰；
- 硬件相关代码和业务代码是否隔离；
- 中断、驱动、业务逻辑之间是否边界明确；
- 初始化顺序是否稳定可控；
- 是否方便后续替换芯片、板卡、通信接口或业务模块；
- 当项目规模变大时，是否还能低风险地维护和定位问题。

下面给出一套更适合长期演进的裸机分层架构思路。

## 设计目标

一个健壮的裸机工程，建议同时追求以下目标：

### 1. 业务逻辑不直接依赖 HAL

`HAL_GPIO_WritePin()`、`HAL_UART_Transmit()` 这类接口很方便，但如果业务层到处直接调用 HAL，那么后续会出现几个问题：

- 板级差异会直接污染业务代码；
- 更换驱动实现时改动面很大；
- 单元测试几乎无法做；
- 中断逻辑、轮询逻辑和业务状态机会耦合在一起。

所以更稳妥的做法是：

- HAL 只作为芯片适配层；
- 项目自己的驱动接口应高于 HAL 一层；
- 业务层只依赖“项目内部驱动接口”。

### 2. 分层必须有单向依赖

如果 `app` 可以随意调用 `drivers`、`bsp`、`hal`、`startup` 里的任意内容，那么即使目录很多，也不算真正的架构。

更合理的原则是：

- 上层可以依赖下层；
- 下层不能反向依赖上层；
- 同层模块尽量通过接口协作，而不是直接互相读写内部状态。

### 3. 中断上下文与主循环上下文必须分开设计

裸机项目最容易失控的地方之一，就是把太多逻辑塞进中断里。

更稳妥的原则是：

- ISR 只做最小工作；
- 中断里尽量只做“采样、搬运、置位、时间戳记录、快速应答”；
- 复杂协议解析、状态机推进、错误恢复，放到主循环或服务层处理。

### 4. 板级差异和产品差异要可隔离

对于长期维护的工程，通常至少会出现下面两类变化：

- 同一套软件适配不同板卡；
- 同一块板卡跑不同产品配置。

因此从一开始就应该预留：

- `board` 级别的硬件资源定义；
- `config` 级别的产品功能开关；
- 驱动和业务之间稳定的抽象接口。

### 5. 中断与主循环协作图

```text

  ↓

  ├─ 收数据到缓冲区
  ├─ 清中断标志

              ↓

```

推荐分工：

- 中断里做“快动作”；
- 主循环里做“重逻辑”；
- 驱动层和服务层负责把两种上下文接起来。

## 推荐的分层模型

结合 STM32 裸机工程经验，我更推荐使用下面这套自下而上的层级。

```text
工具链与启动层
  -> 芯片适配层
    -> 板级支持层
      -> 设备驱动层
        -> 服务与组件层
          -> 应用层
```

下面逐层说明。

### 1. 工具链与启动层

这一层对应当前仓库里的：

- `project/`
- `Libraries/startup/`
- `Libraries/cmsis/`
- `Libraries/stm32f1/`
- `Libraries/hal/`

它的职责应该被严格限定为：

- 提供启动入口；
- 提供链接布局；
- 提供芯片寄存器定义；
- 提供厂家 HAL/LL；
- 提供基础构建能力。

这一层的原则是：

- 尽量少改动厂家原始代码；
- 项目定制逻辑不要散落在 vendor 文件中；
- 所有项目特有逻辑，尽量放到项目自己的适配层里。

换句话说，这一层是“基础设施”，不是“项目逻辑”。

### 2. 芯片适配层

这一层是很多裸机项目里最容易缺失、但实际上非常关键的一层。

它负责把“厂家 HAL/寄存器接口”转换成“项目内部统一可用的底层能力”。它通常包括：

- 时钟初始化；
- 中断优先级规划；
- SysTick 或其他时间基准；
- GPIO/UART/SPI/I2C/ADC/TIM 的底层句柄绑定；
- DMA 通道与外设映射；
- 芯片相关的 `msp` 初始化；
- 临界区、关中断、读系统 Tick 这类底层能力封装。

这一层与 HAL 的关系可以理解为：

- HAL 是厂商接口；
- 芯片适配层是“项目对厂商接口的收口点”。

这样做的好处是：

- 项目其他层不需要知道 HAL 的具体细节；
- 后续如果从 HAL 切到 LL，或者从轮询改成 DMA，中上层接口可以尽量不变。

### 3. 板级支持层 BSP

这一层负责表达“这块板子长什么样”，而不是“业务想做什么”。

它应该解决的问题包括：

- LED 接在哪个 GPIO；
- 按键是高电平有效还是低电平有效；
- UART1 接的是调试串口还是通信串口；
- SPI Flash 挂在哪组引脚；
- 某一路 ADC 对应哪个传感器；
- 板载电源使能脚、蜂鸣器脚、继电器脚、片选脚分别是什么。

BSP 的核心职责是把**板级资源表**固定下来。

推荐做法是：

- 所有引脚、通道、外设实例都由 BSP 统一命名；
- 驱动层通过 BSP 取资源；
- 应用层不要直接写 `GPIOA PIN5` 这类硬编码。

如果没有 BSP，后面几乎一定会出现“业务代码里到处写引脚号”的问题，维护成本会很高。

### 4. 设备驱动层 Drivers

驱动层的职责是把底层外设抽象成稳定、可复用、与业务无关的模块。

例如：

- `drv_uart`
- `drv_spi`
- `drv_i2c`
- `drv_adc`
- `drv_pwm`
- `drv_flash`
- `drv_button`
- `drv_led`

这层有几个重要原则。

#### 驱动只关心设备行为，不关心业务语义

例如一个按键驱动应该提供：

- 初始化；
- 采样；
- 消抖；
- 短按/长按事件输出。

但它不应该知道：

- “按下按键是否意味着进入设置模式”；
- “这个按键是不是电机启动键”。

这些属于应用层语义，不属于驱动层。

#### 每个驱动模块要有明确接口

推荐每个驱动模块至少包含：

- 一个公共头文件；
- 一个源文件；
- 必要时配一个 `port` 或 `cfg` 文件；
- 所有内部状态尽量 `static` 私有化。

例如：

```text
drivers/uart/
  drv_uart.h
  drv_uart.c
  drv_uart_port.h
```

其中：

- `drv_uart.h` 提供项目内部统一接口；
- `drv_uart.c` 实现发送、接收、缓冲区、回调管理；
- `drv_uart_port.h` 负责与具体芯片实例映射。

#### 驱动层要区分同步接口和异步接口

一个成熟的驱动层应避免“所有接口都阻塞”或“所有接口都中断回调乱飞”。

更好的做法是统一约定：

- 哪些接口是阻塞调用；
- 哪些接口是非阻塞启动；
- 哪些事件通过轮询取得；
- 哪些事件通过回调或消息上报。

接口风格统一后，应用层会稳定很多。

### 5. 服务与组件层 Services / Components

这是裸机工程从“小 demo”走向“可维护工程”的关键一层。

这一层不直接操作具体寄存器，也不直接表达最终业务，而是承载那些“跨多个驱动、被多个业务复用”的通用能力，例如：

- 软件定时器；
- 事件分发；
- 环形缓冲区；
- 协议编解码；
- 命令解析；
- 参数存储；
- 故障管理；
- 日志与诊断；
- 状态机框架；
- 电源管理；
- 通信会话管理。

它和驱动层的区别是：

- 驱动层偏“设备”；
- 服务层偏“机制”。

例如：

- `drv_uart` 负责串口收发；
- `svc_console` 负责把串口包装成命令行服务；
- `svc_modbus` 负责把串口包装成 Modbus 通信服务；
- `svc_log` 负责把日志通过某个输出端发出去。

如果没有这层，很多业务代码会直接长在串口中断和驱动回调里，后面会很难维护。

### 6. 应用层 App

应用层只做一件事：

**表达产品行为。**

也就是说，这一层关心的是：

- 设备开机后先做什么；
- 不同状态之间怎么切换；
- 传感器数据达到什么阈值后触发什么动作；
- 某种通信命令到来后怎么改变系统行为；
- 系统故障时进入什么保护模式。

推荐应用层由以下内容组成：

- 应用初始化；
- 主状态机；
- 功能子状态机；
- 周期任务调度入口；
- 业务事件处理逻辑。

应用层不要做的事包括：

- 直接写 GPIO；
- 直接操作 HAL 句柄；
- 在多个文件中零散保存跨模块共享状态；
- 把协议解析和业务动作混在一起。

## 针对当前模板的演进建议

考虑到当前仓库结构还比较简单，不一定要一步到位大改目录。更务实的方式是分阶段演进。

### 第一阶段：在现有模板上补出工程骨架

目标是把当前 `user/` 中的“所有用户代码”拆成更清晰的角色：

- 保留 `user/main.c` 作为启动入口；
- 把 `stm32f1xx_it.c` 保留为统一中断入口；
- 把 `stm32f1xx_hal_msp.c` 保留为 HAL 适配入口；
- 新增 `app/`、`drivers/`、`services/`、`board/`；
- CMake 不再只依赖 `user/*.c`，而是显式纳入新目录源码。

这一阶段最重要的变化，不是功能变多，而是“职责开始分离”。

### 第二阶段：收紧 HAL 的暴露范围

建议逐步做到：

- `app/` 不直接包含 HAL 头文件；
- HAL 句柄尽量只在 `platform/` 或驱动实现文件内部出现；
- 应用层只看项目自己的接口头文件。

这一步完成后，项目的可测试性和可替换性会明显提升。

### 第三阶段：形成事件驱动主循环

裸机项目最怕主循环变成“什么都扫一遍的大泥球”。更好的方式是：

- 把周期采样任务拆出来；
- 把事件和状态变化拆出来；
- 把 ISR 产生的事件通过标志位、环形缓冲区或事件队列交给主循环；
- 主循环只做调度和状态推进。

典型主循环可以是：

```text
while (1)
{
    platform_poll();
    drivers_poll();
    services_poll();
    app_poll();
    watchdog_feed();
}
```

或者按时间片调度：

```text
1ms task
5ms task
10ms task
100ms task
background task
```

关键不是形式，而是：

- 调度入口统一；
- 周期行为可追踪；
- 不让业务逻辑散落在任意位置。

## 必须明确的依赖规则

一个高可维护工程，建议在文档和代码审查中都强制以下规则。

### 1. 依赖方向

推荐依赖关系如下：

```text
app -> services -> drivers -> board/platform -> HAL/CMSIS
common 可被多层复用，但不能反向依赖 app
```

禁止出现：

- `drivers` 依赖 `app`
- `services` 依赖具体业务文件
- `app` 直接操作 HAL 句柄
- BSP 里出现业务判断

### 2. 中断规则

推荐强制以下约束：

- ISR 中不做复杂协议解析；
- ISR 中不调用长时间阻塞函数；
- ISR 中不直接推进大型业务状态机；
- ISR 与主循环共享的数据必须经过明确设计；
- 多字节共享数据访问要考虑临界区保护。

更具体地说，ISR 最适合做：

- 读取寄存器状态；
- 搬运数据到缓冲区；
- 记录时间戳；
- 置事件标志；
- 通知驱动层或服务层有新数据到达。

### 3. 全局变量规则

很多裸机项目后期难维护，根源就是全局变量失控。

建议：

- 模块内部状态默认 `static`；
- 跨模块共享通过接口访问；
- 必须共享的状态集中管理；
- 所有 `extern` 变量都要有明确拥有者；
- `volatile` 只用于真正有并发语义的对象，不要滥用。

### 4. 配置规则

建议把配置拆为三类：

- 芯片/目标配置；
- 板级资源配置；
- 产品功能配置。

不要把下面这些东西混在一个大头文件里：

- 时钟配置；
- 引脚映射；
- 功能开关；
- 通信协议参数；
- 默认阈值；
- 调试选项。

配置分层后，切板卡、切产品、切编译选项都会容易得多。

## 初始化顺序建议

一个健壮裸机工程，初始化顺序最好是“固定、分阶段、可检查”的。

推荐顺序如下：

```text
Reset_Handler
  -> SystemInit
  -> main
    -> platform_init()
    -> board_init()
    -> drivers_init()
    -> services_init()
    -> app_init()
    -> enter main loop
```

也可以把它理解成下面这张更直观的阶段图：

```text
┌──────────────┐
│ platform_init│ 先把“芯片能稳定跑起来”解决
└──────┬───────┘
       ▼
┌──────────────┐
│  board_init  │ 再把“板子资源准备好”解决
└──────┬───────┘
       ▼
┌──────────────┐
│ drivers_init │ 再把“设备可用”解决
└──────┬───────┘
       ▼
┌──────────────┐
│services_init │ 再把“通用机制可用”解决
└──────┬───────┘
       ▼
┌──────────────┐
│   app_init   │ 最后再进入“产品行为”初始化
└──────────────┘
```

各阶段职责建议如下：

- `platform_init()`：时钟、Tick、中断分组、基础底层能力；
- `board_init()`：板级引脚、供电使能、基础外设资源上电；
- `drivers_init()`：UART、SPI、ADC、PWM、Flash 等具体驱动；
- `services_init()`：日志、参数、协议、软件定时器、事件机制；
- `app_init()`：业务状态机和功能模块。

这样做的好处是：

- 初始化失败时容易定位；
- 上电顺序可控；
- 后续新模块知道自己该放在哪一层。

## 裸机工程里最值得重视的几个机制

如果只谈目录分层而不谈这些机制，工程还是会很容易失控。

### 1. 错误处理与断言

建议至少有以下能力：

- `ASSERT()` 或 `PROJECT_ASSERT()`；
- 错误码定义；
- 关键初始化失败的统一处理入口；
- Debug 版本下的断言停机；
- Release 版本下的故障降级策略。

比起“出了问题一律 while(1)”，更好的做法是分类处理：

- 致命错误停机；
- 可恢复错误重试；
- 非致命错误记录并上报。

### 2. 看门狗策略

健壮性很大一部分来自看门狗，而不是只来自代码风格。

建议：

- 在系统稳定后开启看门狗；
- 喂狗动作集中管理；
- 不允许任意模块私自喂狗；
- 喂狗应建立在关键任务都正常执行的前提下。

这样看门狗才能真正反映“系统活着”，而不是“某个循环还在跑”。

### 3. 日志与诊断

即使是裸机项目，也建议尽早统一：

- 调试日志接口；
- 故障码；
- 版本号；
- 编译时间和 Git 版本信息；
- 启动原因、复位原因记录。

后续现场定位问题时，这些能力会非常有价值。

### 4. 状态机

当业务开始变复杂时，最稳的写法通常不是“到处 if/else”，而是显式状态机。

推荐：

- 把系统主流程建成主状态机；
- 把复杂子功能拆成子状态机；
- 状态切换由事件触发；
- 所有状态和事件命名清晰可追踪。

这样代码会比散乱逻辑更容易维护和调试。

## 构建系统建议

基于当前模板的情况，我对构建层有几点比较明确的建议。

### 1. 不建议长期依赖宽泛的 GLOB 收集所有源码

模板阶段这样做很方便，但项目变大后会带来几个问题：

- 构建输入不够显式；
- 新增无关文件可能触发重新生成；
- 很难知道某个目标到底依赖哪些源码；
- 容易把不该参与编译的文件也纳入进来。

更推荐的方式是：

- 显式列出各层源码；
- 或按模块维护子目录 `CMakeLists.txt`；
- 每一层通过自己的 target 或 source list 暴露内容。

### 2. HAL 模块要逐步收缩

当前模板为了通用性，打开了很多 HAL 模块，这在起步阶段可以接受。

但一旦进入具体产品开发阶段，建议同步做两件事：

- 关闭不使用的 HAL 宏；
- 停止编译明显无关的 HAL 源文件。

这样可以减少编译时间，也有助于降低误用风险。

### 3. 生成不同构建配置

后续建议区分至少两类构建：

- `Debug`
- `Release`

如有需要还可以再增加：

- `Factory`
- `Test`

不同配置中可以切换：

- 优化等级；
- 断言开关；
- 日志级别；
- 是否启用测试接口；
- 是否保留额外诊断输出。

## 推荐的模块设计约定

为了让工程长期可维护，建议每个模块遵守统一风格。

### 1. 文件命名

推荐统一前缀，便于快速识别职责：

- `bsp_`：板级资源
- `drv_`：驱动
- `svc_`：服务
- `app_`：应用
- `plat_`：平台适配
- `util_`：通用工具

### 2. 模块接口

每个模块建议至少具备以下元素中的若干项：

- `init`
- `deinit`
- `poll`
- `irq_handler` 或 `on_irq`
- `get/set`
- `process`

不要让同类模块接口风格差异过大，否则后续维护成本会迅速上升。

### 3. 模块职责单一

例如：

- `drv_uart` 只做串口；
- `svc_log` 只做日志；
- `app_motor` 只做电机业务逻辑。

不要出现：

- 一个模块同时负责串口收发、协议解析和业务命令执行；
- 一个文件里同时放 BSP、驱动、业务状态机三种逻辑。

## 一个更适合本项目的总体判断

回到当前这个 STM32F103C8T6 模板，我认为它最适合的演进方向不是“继续把功能都堆到 `user/` 里”，而是逐步变成下面这个结构关系：

```text
Libraries/ + project/
    作为稳定底座

platform/ + board/
    作为硬件收口层

drivers/ + services/
    作为可复用功能层

app/
    作为唯一承载产品行为的层
```

其中最重要的思想是：

- **HAL 不是项目架构本身，它只是项目底层依赖。**
- **BSP 和驱动层不是业务层的附庸，它们应该先被设计清楚。**
- **应用层应该尽量表达“行为”和“状态”，而不是表达“寄存器”和“引脚”。**

## 结论

一个健壮、可维护性高的裸机工程，核心不是目录多，而是下面这些原则是否真正落地：

- 分层清晰，依赖单向；
- 业务不直接耦合 HAL；
- 板级资源有统一收口；
- 中断只做最小必要工作；
- 驱动、服务、应用各司其职；
- 初始化顺序固定；
- 错误处理、日志、看门狗、状态机这些“工程化能力”被认真对待；
- 构建系统和配置系统能够支撑项目长期演进。

如果按这个思路推进，那么当前这个 STM32F103C8T6 模板完全可以从一个“可运行模板”，成长为一个“适合长期维护的裸机产品工程”。
